# Repository Split Design

## Context

Resume Tailor is currently a detached private repository with a monorepo-shaped
layout:

- `apps/backend` contains the FastAPI backend, Python package metadata, `uv`
  lockfile, and backend tests.
- `apps/frontend` contains the Next.js frontend, npm package metadata, UI code,
  i18n messages, and frontend tests.
- The repository root contains a deployment stack that builds and runs both
  applications together: `Dockerfile`, `docker-compose.yml`, `docker/start.sh`,
  `railway.json`, and `scripts/deploy-railway.sh`.

The next phase should not preserve the current deployment model. The backend is
expected to undergo major transformation, and the current combined Docker/Railway
stack would add migration drag without representing the desired future shape.

## Decision

Split the project into two clean private repositories:

- `tailoring-engine`
- `resume-studio`

Use clean imports rather than history-preserving subtree splits. The existing
repository remains a source reference and archive until the split is stable.

## Product Boundary

The split is defined by data flow:

```text
tailoring-engine -> API/OpenAPI/responses -> resume-studio
```

`tailoring-engine` produces the contract and responses. `resume-studio` consumes
what the engine sends. Avoid ownership matrices or monorepo-style package
governance in the new repositories.

## `tailoring-engine`

Start from:

- `apps/backend/app/`
- `apps/backend/tests/`
- `apps/backend/pyproject.toml`
- `apps/backend/uv.lock`
- backend-specific agent instructions, rewritten for the new repo root
- backend/API docs that remain useful after the split

The backend repository should own the API contract. OpenAPI generated by the
engine is the source of truth for frontend integration. The initial import may
preserve current backend code, but the design assumes a major rewrite is likely.

## `resume-studio`

Start from:

- `apps/frontend/app/`
- `apps/frontend/components/`
- `apps/frontend/hooks/`
- `apps/frontend/lib/`
- `apps/frontend/messages/`
- `apps/frontend/public/`
- frontend package and tool configuration files
- frontend-specific agent instructions, rewritten for the new repo root
- UI, design, i18n, and frontend workflow docs that remain useful after the
  split

The frontend should call an externally configured engine URL. It should not
inherit the current combined-image assumption that backend and frontend are
always started by the same process manager.

## Contract Strategy

In the first split pass, `resume-studio` may keep its hand-written API helpers
to reduce migration risk. Before release, those helpers must be checked against
the engine contract.

After the backend transformation stabilizes, prefer generated or checked
TypeScript types from `tailoring-engine` OpenAPI. Do not introduce a third shared
contract repository until there is a real release coordination need.

## Files Not To Preserve

Do not carry the current deployment stack into either new repository:

- `Dockerfile`
- `docker-compose.yml`
- `docker/start.sh`
- `railway.json`
- `scripts/deploy-railway.sh`

Do not copy public upstream community files into the private repos unless they
serve the new private workflow.

## Plan And Design Files

Preserve non-superpowers plan and design files when they describe product,
domain, UI, API, or architecture decisions that are still useful. Route them to
the repository that will use them.

Superpowers specs and plans are historical working artifacts. Keep them in the
archive repository unless a specific file is needed as source material for a new
repo-local plan.

## Validation

Each new repository must be independently valid.

For `tailoring-engine`:

```bash
uv sync --extra dev
uv run pytest
```

For `resume-studio`:

```bash
npm ci
npm run lint
npm run test
npm run build
```

After both repositories exist, run a cross-repo smoke test: start the engine,
configure studio with the engine URL, verify engine health, and exercise one
real resume flow.

## Open Questions

- What private GitHub owner should hold `tailoring-engine` and `resume-studio`?
- Should `resume-studio` initially call the engine directly, or should a future
  edge/proxy layer own same-origin routing?
- Which non-superpowers design files should be copied verbatim, and which should
  be rewritten into slimmer repo-local docs?
